### モックで画面化する Must ストーリー優先順

`stories.md` の MoSCoW Must（10 個）のうち、モックで特に体験を磨くべき 4 つ:

- **Story 1.3** 初回サジェスチョン受領 — 相棒トーンの第一印象
- **Story 2.4** Phase 4「やっといたよ」音声報告 — Web Speech API で疑似体験、エモーショナルピーク
- **Story 4.3** 完全委譲のドラマ — 確認ダイアログの重み、テーマクライマックス
- **Story 5.5** 自己決定能力スコア時系列 — 堕落の可視化、プロダクトの代名詞

残りの Must（1.1 アカウント登録、1.2 カテゴリ選択、2.1〜2.3 Phase 1〜3、5.1 現在フェーズ）も画面化するが、磨き込みは上記 4 つを優先。

## 1. 目的

### 1.1 なぜモックを作るのか
- **「リューク的相棒トーン」** の文言が画面に乗ったときの温度感を確認する（文字で読むのと触る体験は違う）
- **「委ねるボタン」** の押下ドラマ性（罪悪感＋解放感）を実体験で検証する
- **音声 UI のタイミング** を Web Speech API で疑似体験し、Phase 4「やっといたよ」の違和感／自然さを確認する
- **Phase 1〜4 の遷移体験** をハードコードのモック応答で見せ、ストーリー全体の流れを磨く
- **Application Design 質問（C-1〜C-8）への回答精度向上** に活用する

### 1.2 何を作らないか
- 本番コード（Construction フェーズで書く）
- 実 AWS 連携（Bedrock、Polly、Cognito、DynamoDB は使わない）
- バックエンド（フロントエンドのみ）
- テスト（Discovery 用なので不要）

---

## 2. スコープ（L2: Frontend-only モック）

### 含めるもの
- **画面 3 枚程度**:
  - サジェスチョン画面（Phase 1〜3 のいずれか）
  - 「もう連絡のことは考えたくない」完全委譲ドラマ（確認ダイアログ）
  - ミラー画面（堕落可視化、自己決定能力スコアグラフ含む）
- **委ねるボタン 3 種** の挙動（単発委譲 / フェーズ昇格 / 完全委譲）
- **音声 UI の疑似体験** — Web Speech API で TTS。「お母さんに『元気だよ』返信しといたから」を実際に喋らせる
- **ハードコードの応答データ**（連絡カテゴリ用 LINE 親返信文案 / 人間関係カテゴリ用 飲み会断り文案）
- **リューク的相棒トーンの文言例** をいくつか試して比較

### 含めないもの
- ログイン・サインアップ
- リアル LLM 呼び出し
- DB 永続化（State はメモリ or localStorage）
- ルーティング以外の認証
- 複数ユーザー対応

## 3. 技術選定

| 要素 | 選定 | 理由 |
| --- | --- | --- |
| ビルドツール | **Vite** | 起動が速い、設定がシンプル |
| UI フレームワーク | **React + TypeScript** | チームが慣れている（B-3 = X 回答） |
| スタイリング | **Tailwind CSS** または **vanilla CSS** | 速度優先、好みに応じて選択 |
| 音声合成 | **Web Speech API（SpeechSynthesis）** | ブラウザ標準、Polly の代替として疑似体験 |
| 状態管理 | **React の useState / useReducer のみ** | 永続化不要 |
| ホスティング | **ローカル開発のみ**（`npm run dev`） | デプロイ不要 |

## 6. Application Design 質問との連携

モック制作を通じて、以下の質問への回答精度を上げる:

| Application Design 質問 | モックで何を検証するか |
| --- | --- |
| C-3 API スタイル | フロントから必要なエンドポイント数を可視化 → REST or GraphQL の判断材料 |
| C-4 Phase 4 自律実行トリガー | デモ用ボタンと EventBridge スケジュールのどちらが体験に必要か |
| C-5 音声通知デリバリ | Web Speech API でフォアグラウンド再生を試す → WebSocket / Push の必要性判定 |
| C-7 フロントアーキ | SPA でモック作成 → 本番も SPA で十分かの感触 |
| C-8 外部送信モック境界 | モックは送信なしで成立 → 「完全モック」の選択を確認 |

**運用**: モックで気づいた知見を `application-design-plan.md` の質問回答時に反映する。モック完了（5/3）まで Application Design 質問の回答は **保留** することを推奨する（前回の回答で 3 = 並行）。回答中の方は、モックの結果次第で調整する前提で記入してよい。

## 7. 知見の還元ルール

モックでの発見は **以下の手順で正本ドキュメントに反映** する:

1. モックを触って気付いたことを `discovery-mock/README.md` の「発見事項」セクションに箇条書きで記録
2. 5/3 のチーム振り返りで知見を共有
3. 知見を `application-design-plan.md` の質問回答に反映
4. または `requirements.md` / `stories.md` に追記が必要なら記録（ペルソナ追加の必要性、文言の決定、UX フローの修正など）
5. **モックコード自体は Construction フェーズで参照しない**（捨てる前提）
