## 2. ストーリー分割アプローチ

### 計画ステップ
- [ ] アプローチを確定し、その方針で全ストーリーを分割
- [ ] 主要 Epic（大きな塊）を定義
- [ ] 各 Epic の配下にストーリーを配置
- [ ] ストーリー総数の目処を立てる

### Question A-1: ストーリー分割アプローチ

複数のアプローチが選べます。本プロダクトには「Phase 1〜4 の遷移」「カテゴリ別」「エージェント別」「UI チャネル別」などの軸が存在します。

A) **User Journey-Based（ユーザージャーニー軸）** — 連絡カテゴリで Phase 1 → 2 → 3 → 4 を辿る一連のジャーニーをストーリー化。人間関係も同様に。**ハッカソン審査の "ストーリー性" には強い**
B) **Feature-Based（機能軸）** — サジェスチョン提示、選択ログ、委譲ボタン、フェーズ判定、自律実行、ミラー…など機能単位
C) **Persona-Based（ペルソナ軸）** — ペルソナ別にストーリーをグループ化
D) **Domain-Based（ドメイン軸）** — 連絡ドメイン／人間関係ドメイン／可視化ドメインで分割
E) **Epic-Based 階層（Epic > Story > Sub-story）** — Phase 1〜4 を 1 つの Epic とし、その配下にストーリーを並べる
F) **ハイブリッド: User Journey + Epic**（推奨候補） — Epic として「連絡カテゴリの Phase 1→4」「人間関係カテゴリの Phase 1→4」「ミラーでの可視化」を立て、その配下にユーザージャーニーに沿ったストーリーを並べる
X) その他（下の `[Answer]:` タグの後に詳述してください）

[Answer]: F

## 8. 必須成果物（INVEST 準拠）

以下は MANDATORY ARTIFACTS。Part 2 Generation で必ず生成する:

- [ ] `aidlc-docs/inception/user-stories/personas.md` — ペルソナ集
- [ ] `aidlc-docs/inception/user-stories/stories.md` — ユーザーストーリー集（INVEST 準拠 + 受け入れ条件 + ペルソナマッピング）

INVEST 原則:
- **I**ndependent（独立性）— 他ストーリーに依存しない
- **N**egotiable（交渉可能）— 詳細はチームで決める余地を残す
- **V**aluable（価値あり）— ユーザー価値が明確
- **E**stimable（見積もり可能）— サイズ感が把握できる
- **S**mall（小さい）— 1 スプリント／数日で実装可能
- **T**estable（テスト可能）— 受け入れ条件で検証できる
