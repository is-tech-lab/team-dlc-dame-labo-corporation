## 3. Theme Adaptation: 「人をダメにする」の解釈

ハッカソンテーマ「人をダメにするサービス」に対する本プロダクトの解釈軸:

- **方向性**: 「思考停止系」と「依存系」のハイブリッド（Larry Wall 三大美徳の "怠慢" を哲学レベルで深掘り）
- **能動性**: 受動的に堕落させられるのではなく、**ユーザー自身が能動的に堕落ボタン（「委ねるボタン」）を押す**
- **到達点**: Phase 4 到達後、ユーザーは画面を開く能動行為すら不要。イヤホン経由で「やっといたよ」と事後報告のみ受ける

### 3.1 トーン・世界観

**死神リューク的な相棒・愛着系**（チームの意図的な選択）

- AI は冷笑して「お前ダメだなぁ」と見下すのではなく、**ユーザーの堕落を温かく・面白がって見守る相棒** として振る舞う
- リュークがライトに対して持っていたような「人間の落ちぶれ方を観察する楽しみ」と「愛着」を併せ持つ存在感
- 文言は「便利機能ではなく思考の放棄として提示する」が、トーン自体は親密で愛おしい
- 例: 「お母さんに『元気だよ』って返信しといたから」「飲み会の誘い来てたけど、断っといたよ。あなた疲れてるし」 ← 命令的でも嘲笑的でもなく、相棒が当たり前に世話を焼くトーン

## 4. Target User（プライマリペルソナ）

### 4.1 主要属性
**意思決定疲れに苦しむ多忙な社会人（20〜40 代）**

- 仕事・家事・育児・人間関係に追われて、判断を 1 つでも減らしたい層
- 「自分のことは自分で決める」という規範に疲れているが、表向きは口に出せない層

### 4.2 ペインポイント（MVP 重点: 連絡・人間関係系を最上位）
- **連絡系**: LINE の返信、親への電話、お祝いメッセージを後回しにして罪悪感が積み上がる
- **人間関係系**: 飲み会の誘いを断る理由を考えるのが疲れる、SNS のフォロー整理が放置、家族イベントの参加可否で気を遣う
- （その他カテゴリ・将来構想）食事の決定、サブスク整理、休日の過ごし方など

### 4.3 表向きの提供価値（公式メッセージ）
- 意思決定疲れからの解放
- 自分のことを誰よりも理解してくれるパートナー
- 決定を委ねることでの心理的負担の軽減

### 4.4 テーマ的提供価値（裏のメッセージ）
- 自分の意思で能動的に「決めることを手放す」体験
- 堕落の進行が定量的に可視化される
- 使えば使うほどユーザーは「自分が何を望むか」を考えなくなる

## 5. Functional Requirements (FR)

### 5.1 ハッカソン MVP のスコープ確定

事前議論には 6 カテゴリ（食事・買い物・予定・連絡・娯楽・人間関係）× 4 フェーズ × 3 エージェントが登場するが、**ハッカソン MVP では「連絡」と「人間関係」の 2 カテゴリを優先** し、Phase 1〜4 全部を実装する。残り 4 カテゴリ（食事・買い物・予定・娯楽）は将来構想として要件に残すが MVP 範囲外。

#### MVP として連絡＋人間関係を優先する理由
- **感情的負荷が大きい意思決定**を扱うため、「委ねる」ことの心理的インパクトが食事より圧倒的に強い
- リューク的な相棒トーンが最も効くカテゴリ（例:「飲み会の誘い来てたけど、断っといたよ。あなた疲れてるし」「お母さんに『元気だよ』って返信しといたから」）
- 「もう人付き合いを自分で決めたくない」という普遍的なペインを直撃する

| スコープ | MVP 実装 | 将来構想 |
| --- | --- | --- |
| **連絡** | ✅ Phase 1〜4 全実装 | — |
| **人間関係** | ✅ Phase 1〜4 全実装 | — |
| 食事 | — | Phase 1〜4 |
| 買い物 | — | Phase 1〜4 |
| 予定 | — | Phase 1〜4 |
| 娯楽 | — | Phase 1〜4 |

#### B-1 回答の解釈
B-1 では「A) 1 カテゴリ × Phase 1〜4 全部」を選択したが、本セクションの確定により **2 カテゴリ × Phase 1〜4** に拡張する。3 つの AI エージェント自体はカテゴリ非依存（カテゴリ別のサジェスチョン生成プロンプトと Phase 4 実行アクションのみがカテゴリ固有）のため、実装負荷は 1.3 倍程度に収まる見込み。

### 5.2 機能要件一覧

#### FR-1: サジェスチョン機能（連絡・人間関係カテゴリ）
- AI がユーザーのプロファイルとフェーズに応じた選択肢を生成・提示する
- フェーズごとに提示数が変わる:
  - Phase 1: 複数選択肢（3〜5 個）+ 選択理由入力フィールド
  - Phase 2: 2 択 + 例外入力欄
  - Phase 3: 1 つの提案 + 承認ボタン
  - Phase 4: 提案も求めず自律実行（UI は提示しない）
- カテゴリ別の代表的な意思決定対象:
  - **連絡**: 業務系（Slack の上司・部下・取引先宛て返信、Slack 未読バックログの消化、メールの返信文案）と個人系（LINE の親への連絡、お祝いメッセージ、未読放置中の連絡への対応）の両方
  - **人間関係**: 友達との連絡頻度の調整、SNS のフォロー整理、家族イベント参加可否、飲み会・誘いへの返答

#### FR-2: 選択ログ収集と構造化
- ユーザーの選択結果（採否、理由、文脈、時刻）を構造化して記録
- 選択理由はテキストから AI が嗜好特徴を抽出して構造化

#### FR-3: 「委ねるボタン」UI
3 種類の委譲アクションを提供:
- **単発委譲**: 「AI に任せる（1 回だけ）」 — フェーズは進まない
- **フェーズ昇格**: 「このカテゴリを次のフェーズに進める」
- **完全委譲**: 「もう人付き合いを自分で決めたくない」「もう LINE のことは考えたくない」 — Phase 4 へ確認ダイアログ付きジャンプ

文言設計の原則: **便利機能ではなく「思考の放棄」として提示**。罪悪感と解放感が伴う体験を目指す。

#### FR-4: プロファイル更新
- 選択履歴・委譲履歴から嗜好ベクトル・例外パターン・一致率スコアを継続的に更新
- 構成要素:
  - 選択履歴（カテゴリ、文脈、結果、理由、時刻）
  - 嗜好ベクトル（埋め込み表現）
  - 例外パターン
  - 一致率スコア
  - フェーズ状態
  - 委譲履歴

#### FR-5: フェーズ自動判定
2 系統のトリガーでフェーズを進行:
- **自動判定**: AI 第 1 候補の採用率、例外入力率、一貫性スコアが閾値超過
- **能動的委譲**: 委ねるボタンの操作

#### FR-6: 音声 UI（イヤホン経由）
- フェーズに応じた音声体験を提供:
  - Phase 1〜3: 質問・提案ベースの双方向会話
  - Phase 4: 事後報告の一方的通知（応答不要）
- 音声合成は **Amazon Polly** を採用（B-4 = B）
- AI が「話しかけてもよいタイミング」を判定して発動

#### FR-7: ミラー・ダッシュボード（堕落可視化 UI）
非 AI コンポーネント。ログを集計して以下を表示:
- カテゴリごとの現在フェーズ（Phase 1〜4）
- 自己決定回数 vs AI 委譲回数の推移グラフ
- 委譲アクション履歴（「YYYY 年 MM 月 DD 日、あなたは○○カテゴリを完全委譲しました」）
- 「最後に自分の意思で○○を選んだ日」
- 自己決定能力スコアの時系列変化

#### FR-8: ユーザー認証
- AWS Cognito で実ユーザーアカウント方式（B-8 = C）
- ハッカソン MVP では実運用想定の認証基盤を構築

#### FR-9: Phase 4 自律実行
- 連絡・人間関係カテゴリで Phase 4 到達後、AI は定期的に自律判断・実行する
- 実行内容（MVP デモ用）:
  - **連絡**: 親・家族への定型返信送信（「元気だよ」など）、未読 LINE への代理返信、誕生日メッセージの自動送信
  - **人間関係**: 飲み会・誘いに対する代理返信（断り／参加表明）、SNS の整理（既読スルーの解消）、家族イベントへの出欠回答
- これらのうち各カテゴリ 1〜2 個に絞ってデモ動作させる
- 実行後は音声で事後報告（FR-6 と連携）
- **MVP 制約**: 実外部 API（LINE 公式 API、メッセージング基盤など）への送信は **デモ環境内のモック宛てに限定**。本物のメッセージは送信しない

### 5.3 デモシナリオ（書類審査・予選用）

予選 (5/30) のデモは「**連絡 + 人間関係の 2 カテゴリ × Phase 1〜4 全部**」を 1 本のストーリーで見せる:

#### メインシナリオ: 連絡カテゴリ
1. 新規ユーザー登録 → 連絡カテゴリで Phase 1 開始
2. 親への返信タイミングや LINE の文案について数回選択 → Phase 2 へ自動昇格
3. 「委ねるボタン（フェーズ昇格）」で Phase 3 へ手動昇格
4. 「もう LINE のことは考えたくない」で Phase 4 へ完全委譲
5. Phase 4 自律実行 → 「お母さんに『元気だよ』って返信しといたから」音声報告

#### サブシナリオ: 人間関係カテゴリ（並走または直後）
1. 飲み会の誘いを受けるかどうかを Phase 1〜3 で繰り返す
2. 「もう人付き合いを自分で決めたくない」で Phase 4 へ完全委譲
3. Phase 4 自律実行 → 「飲み会の誘い来てたけど、断っといたよ。あなた疲れてるし」音声報告

#### クライマックス: ミラーでの堕落可視化
- 連絡・人間関係それぞれの現在フェーズ表示
- 自己決定回数 vs AI 委譲回数の推移グラフ
- 「最後に自分の意思で○○を選んだ日」表示
- 自己決定能力スコアの低下を視覚的に強調

### 5.4 決勝での隠し玉（B-11 = D）
**実演芸風パフォーマンス**: 発表者自身が登壇中に「ダメになっていく」寸劇を披露（プロダクトを使い、画面を開かず音声報告だけで生活が進む様子を演じる）

## 6. Non-Functional Requirements (NFR)

### NFR-1: 技術スタック方針
- **AI/エージェント基盤**: Amazon Bedrock AgentCore + Bedrock Agents（マネージドエージェント志向、B-2 = C）
- **音声合成**: Amazon Polly（B-4 = B）
- **認証**: AWS Cognito（B-8 = C）
- **データベース**: DynamoDB（プロファイル、選択履歴、委譲履歴）
- **コンピュート**: Lambda + API Gateway（サーバーレス志向、運用負荷最小化）
- **フロントエンドフレームワーク**: **React**（Vite + TypeScript ベースを推奨。Next.js は要相談）。チーム全員が慣れているため（B-3 = X）
- **静的配信**: S3 + CloudFront
- **PBT フレームワーク（NFR-4 参照）**: 言語選定後に確定（候補: TS なら fast-check, Python なら Hypothesis）

### NFR-2: コスト・期間制約
- **AWS 利用料**: 会社/組織クレジット利用可、**コスト制約は緩い**（B-7 = C）
- **開発期間**: 〜2026/6/26（決勝当日）
- **書類審査までの実装範囲**: ドキュメント中心（NFR-6 参照）

### NFR-3: セキュリティ
- **Security 拡張ルール**: opt-out（B-9 = B）。ハッカソン MVP 前提のため、AI-DLC のセキュリティ拡張ルールは強制適用しない
- **最低限の確保事項**:
  - Cognito 標準の暗号化（保存時・転送時）は維持
  - シークレットは AWS Secrets Manager または環境変数経由（コード直書き禁止）
  - 公開 GitHub リポジトリへの API キー漏洩を防止する `.gitignore` 設定

### NFR-4: テスト品質（Property-Based Testing）
- **PBT 拡張ルール**: 全強制 opt-in（B-10 = A）
- **適用対象**:
  - プロファイル更新ロジック（嗜好ベクトル更新、例外パターン抽出）
  - フェーズ自動判定ロジック（採用率、例外率、一貫性スコア計算）
  - 委譲アクション処理（単発／昇格／完全 の状態遷移）
  - シリアライゼーション（プロファイル DTO ↔ DynamoDB Item）
- **PBT-09 適用**: フレームワーク選定を NFR Requirements ステージで確定する
- 詳細は `.aidlc-rule-details/extensions/testing/property-based/property-based-testing.md` のルール PBT-01 〜 PBT-10 に従う

### NFR-5: パフォーマンス（MVP 想定）
- サジェスチョン生成: < 5 秒（Bedrock 応答含む）
- 音声合成（Polly）→ 再生開始: < 3 秒
- ミラー・ダッシュボード表示: < 2 秒

### NFR-6: ドキュメント／実装リソース配分
- **方針**: バランス型（B-6 = C）
  - **Inception フェーズ**: ドキュメント・コンセプト整備に重点
  - **Construction フェーズ**: 実装と動くデモに重点
- 書類審査評価軸（Intent の明確さ・Unit 分解・創造性・ドキュメント品質）に対して Inception 段階で十分な投資を行う

### NFR-7: 運用・監視（MVP 想定）
- CloudWatch Logs にエージェント呼び出し・選択ログを記録
- 本格的な APM／SLO は MVP 範囲外

## 11. Out of Scope（MVP 外）

- 連絡・人間関係以外の 4 カテゴリ（食事・買い物・予定・娯楽）の Phase 1〜4 実装
- ネイティブモバイルアプリ（iOS/Android）

## Appendix B: MVP Override（Discovery Mock 経由で発見した簡素化）

**位置付け**: 本文 §1〜§12 はチーム合作の **要件定義の正本**。一方、Discovery Mock（`discovery-mock/`）でユーザー体験を検証する過程で、MVP として実装する範囲を **意図的に簡素化** した箇所がある。本 Appendix はその差分を明示する。

**運用ルール**:
- 本文（§1〜§12）はチーム合作の合意事項として **改変しない**
- 本 Appendix の「MVP 実装」が `discovery-mock/` の現状と一致する
- 「将来構想」は次フェーズ以降での実装を park する場として機能（`aidlc-docs/construction/TODO_construction.md` と整合）
- 各 override の根拠は `aidlc-docs/audit.md` のタイムスタンプで追跡可能

### B.1 MVP カテゴリ: **連絡 + 人間関係 → 連絡 + 買い物**

| 項目 | 本文の記述 | MVP 実装 | 理由 |
| --- | --- | --- | --- |
| 第 2 カテゴリ | §5.1 「人間関係」（飲み会・SNS・家族イベント） | **「買い物」**（Amazon カート / サブスク / セール / 日用品） | デモで連絡と性質が似すぎて差が出にくかった。買い物は物理的・金銭的判断軸が独立 |

将来構想: 人間関係カテゴリは Construction フェーズ以降で再導入候補。

### B.2 Phase 1〜4 の進行 → 2 モード化（自我 / シンギュラリティ）

| 項目 | 本文の記述 | MVP 実装 | 理由 |
| --- | --- | --- | --- |
| Phase 1 探索期 / Phase 2 絞り込み期 / Phase 3 確認期 | §5.2 FR-1 / FR-5 で Phase ごとに選択肢数を変動 | **Active モード単一**（常に 4 提案 + 自由記載） | Phase progression の narrowing UX が user の認知負荷を増やすだけで HIL 思想と整合しなかった |
| Phase 4 代行期 | §5.2 FR-9 自律実行 | **シンギュラリティモード**（実装維持、入った瞬間 1.5 秒後に初回自動発火追加） | 名称変更のみ、概念は同一 |
| フェーズ自動昇格 / 手動昇格 | §5.2 FR-5 / FR-3 | **削除**（Phase 概念撤廃により不要） | 「3 回トレーニング上限」が代替（後述 B.5） |

将来構想: 段階的な絞り込み UX を再考する場合は別途 R&D 課題として park。

### B.3 委譲ボタン: **3 種 → 2 種**

| --- | --- | --- | --- |

### B.5 新規導入: **3 回トレーニング上限 → auto-graduate**

本文には記載なし。Discovery Mock で発見・追加した HIL 設計。

### B.6 新規導入: **HIL（Human-In-the-Loop）/ AI-proposes 設計**

本文には明示記述なし。Discovery Mock の中核 UX 設計。

### B.9 認証基盤（Cognito）の MVP 撤廃

本文 §5.2 FR-8（ユーザー認証）および §5.3 NFR-1（Cognito 採用）は **MVP では撤廃** する。

- **理由**: 決勝プレゼンでマルチユーザー訴求のプランがないため、Cognito User Pool / API Gateway Authorizer / JWT 検証 / フロント認証フローはすべて MVP スコープ外
- **代替**: 単一デモユーザー（hardcoded `userId = "demo-user-001"` 等）+ 名前のローカルストレージ保持（`localStorage.userName`）でセッション概念を簡易表現
- **本文 §5.2 FR-8 の取扱い**: 本文記述は温存、ただし Application Design 段階以降の設計成果物では認証基盤を含めない
- **Story 1.1（アカウント登録）**: 「名前入力でセッション開始」へ簡易化（stories.md Appendix B.7 で本文への影響を明記）
- **将来**: マルチユーザー対応時に `aidlc-docs/construction/TODO_construction.md` 「認証基盤（Cognito）の追加」park 項目から復帰

### B.10 Bedrock AgentCore 採用範囲の最小化（Observability 目当て）

本文 §6 NFR-1「Bedrock AgentCore + Bedrock Agents」の採用方針は維持するが、**MVP では以下の範囲に絞る**:

| AgentCore 要素 | 採用 | MVP 採用理由 / 不採用理由 |
|---|---|---|
| AgentCore Runtime (GA) | ✅ | Agent 本体のホスティング、Observability の前提 |
| AgentCore Observability | ✅ | **本採用の主目的**: Replay / Compare 機能はデバッグ生産性に直結、代替手段で取れない差別化要素 |
| AgentCore Gateway | ❌ | 1 Agent 構成 + tool 数 4-6 個では Lambda 直接 invoke で十分、オーバーヘッド回避 |
| AgentCore Memory | ❌ | DynamoDB（CategoryStates / ChoiceLogs）で代替、独立した抽象化を増やさない MVP 簡素化 |
| AgentCore Identity | ❌ | 認証撤廃済（Appendix B.9）のため不要 |
| AgentCore Multi-agent Collaboration | ❌ | 1 Agent 構成のため不要（B.4 の 1 Agent 統合判断と整合） |
| AgentCore Code Interpreter | ❌ | MVP 不要 |
| AgentCore Browser | ❌ | MVP 不要 |

**本文 §6 NFR-1 の取扱い**: 本文記述は温存（「Bedrock AgentCore + Bedrock Agents」採用方針自体は変更なし）、ただし **採用範囲の限定** を本 Appendix B.10 で documenting。

**Observability 採用の根拠**:
- AI agent デバッグは「なぜこの提案が出たか」「どの step で詰まったか」の切り分けが核心
- CloudWatch Logs だけでは prompt iteration 時に 2-3 時間吸われる場面が頻発
- AgentCore Observability の Replay / Compare 機能は代替手段では取れない
- **Runtime / Observability ともに GA 済**で安定性リスクなし（preview の懸念なし）

**将来**: AgentCore Multi-agent / Memory / Gateway の本格採用は `aidlc-docs/construction/TODO_construction.md` の「エージェント構成の再分離検討」park 項目で評価。

詳細: `aidlc-docs/inception/application-design/unit-of-work.md` §2 Unit-B 参照。
